# Problems Encountered

## Overview

This document records the main problems encountered while building the Embedded Desk Terminal v0.

The project was successful, but several problems appeared during the process. Most of them were not caused by the electronic components themselves, but by wiring, build system configuration, file organization, and beginner-level workflow mistakes.

Documenting these problems is important because debugging is part of real engineering work.

---

## Problem 1: Confusion Between Source Folder and Build Folder

### Problem

At the beginning, I was editing and looking at files inside the `build/` directory.

This caused confusion because the `build/` directory contains files generated by CMake, such as:

```text
CMakeFiles/
CMakeCache.txt
Makefile
*.elf
*.uf2
```

These are not the main source files of the project.

### Cause

I opened or worked from the build directory instead of the project root directory.

### Fix

The correct project structure should separate source files from generated build files:

```text
project-folder/
├── CMakeLists.txt
├── blink.c
├── lcd_i2c.c
├── lcd_i2c.h
├── pico_sdk_import.cmake
└── build/
```

The source files should be edited in the project root folder, not inside `build/`.

### Lesson Learned

The `build/` folder is for generated files. The actual source code and CMake configuration should stay outside the build directory.

## Problem 2: Missing `pico_sdk_import.cmake`

### Problem

CMake failed with an error saying that it could not find:

```text
pico_sdk_import.cmake
```

### Cause

The project folder did not contain the `pico_sdk_import.cmake` file.

This file is needed so CMake can locate and use the Raspberry Pi Pico SDK.

### Fix

I searched for the file inside the installed Pico SDK and copied it into the project folder.

The file was located at:

```text
/home/gustavo/embedded/pico-sdk/external/pico_sdk_import.cmake
```

It was copied into the project folder using:

```bash
cp /home/gustavo/embedded/pico-sdk/external/pico_sdk_import.cmake .
```

### Lesson Learned

A Pico SDK project needs access to `pico_sdk_import.cmake`, either by copying it into the project or configuring the environment correctly.

## Problem 3: Confusion Between `main.c`, `blink.c`, `main.uf2`, and `blink.uf2`

### Problem

The project originally used a file called `blink.c`, but some CMake examples expected a file named `main.c`.

This created confusion between:

```text
main.c
blink.c
main.uf2
blink.uf2
```

### Cause

The CMake target name and the actual source file name were not consistent.

For example, if CMake says:

```cmake
add_executable(main
    main.c
)
```

then the output will be based on the target `main`.

But if the source file is actually called `blink.c`, CMake must reference `blink.c`.

### Fix

The project was made consistent by using `blink` as the target and `blink.c` as the main source file:

```cmake
add_executable(blink
    blink.c
    lcd_i2c.c
)
```

This generates:

```text
blink.uf2
```

### Lesson Learned

The CMake target name controls the output file name, while the source file list tells CMake which C files to compile.

Consistency matters.

## Problem 4: `undefined reference to main`

### Problem

During compilation, the linker produced this error:

```text
undefined reference to `main`
```

### Cause

The build system could not find a valid `main()` function in the files being compiled.

This can happen when:

- CMake is compiling the wrong file.
- The source file does not contain `int main(void)`.
- The CMake target references `main.c`, but the actual file is `blink.c`.

### Fix

The `CMakeLists.txt` file was corrected to compile the correct source file:

```cmake
add_executable(blink
    blink.c
    lcd_i2c.c
)
```

The firmware file also contained a valid main function:

```c
int main(void)
{
    // firmware code
}
```

### Lesson Learned

In C projects, the program must contain a valid `main()` function, and CMake must compile the file that contains it.

## Problem 5: LED Wiring Confusion

### Problem

At first, wiring the LED to the Raspberry Pi Pico was confusing.

The main question was how to physically connect:

```text
GP10
resistor
LED
GND
```

### Cause

The Raspberry Pi Pico was not inserted directly into the breadboard, so the wiring had to be done using jumper wires from the Pico headers to the breadboard.

This made the connection less visually obvious.

### Fix

The correct LED wiring was:

```text
GP10 -> resistor -> LED anode
LED cathode -> GND
```

The LED anode is the longer leg, and the cathode is the shorter leg.

A current-limiting resistor was placed in series with the LED.

### Lesson Learned

A GPIO output circuit needs a complete path:

```text
GPIO -> load -> GND
```

For an LED, the resistor is required to limit current and protect the circuit.

## Problem 6: Push Button Placement

### Problem

The tactile push button did not work correctly at first.

### Cause

A four-pin tactile push button can be confusing because some pins are already connected internally.

Usually:

```text
Two pins on one side are connected together.
Two pins on the opposite side are connected together.
```

The button only closes the circuit when opposite sides are used.

### Fix

The button was placed across the center gap of the breadboard.

The correct wiring was:

```text
GP16 -> push button -> GND
```

The firmware enabled the internal pull-up resistor:

```c
gpio_pull_up(BUTTON_PIN);
```

Because of that, the button used active-low logic:

```text
Not pressed -> GPIO reads HIGH
Pressed     -> GPIO reads LOW
```

### Lesson Learned

A four-pin tactile button must be wired using opposite sides, not two pins from the same internally connected side.

## Problem 7: LCD Contrast Adjustment

### Problem

The LCD backlight turned on, but the text was not clearly visible at first.

### Cause

The LCD contrast was not adjusted correctly.

The LCD1602 I2C module has a small potentiometer on the back that controls contrast.

### Fix

The contrast potentiometer was adjusted slowly until the text became visible.

After that, the LCD correctly displayed:

```text
EMBEDDED TERM
LCD OK
```

### Lesson Learned

If an LCD backlight turns on but no characters are visible, the problem may be contrast, not code.

## Problem 8: GitHub Push Authentication

### Problem

After committing the project locally, `git push` failed.

The terminal showed that password authentication was not supported for Git operations.

### Cause

GitHub no longer accepts normal account passwords for HTTPS Git operations.

### Fix

GitHub CLI was used to authenticate properly:

```bash
gh auth login
```

After logging in through the browser, `git push` worked correctly.

### Lesson Learned

GitHub requires modern authentication methods such as GitHub CLI, SSH keys, or personal access tokens.

## Problem 9: Servo Power Considerations

### Problem

The SG90 servo worked correctly when powered from VBUS, but this is something that must be treated carefully.

### Cause

Servos can draw more current than simple components like LEDs or buttons.

If the servo draws too much current, it can cause unstable behavior.

Possible symptoms include:

```text
Servo jitter
Pico reset
USB disconnect
LCD flicker
Random behavior
```

### Fix

For this prototype, powering the SG90 from VBUS worked correctly.

However, for a more robust version, a separate 5V power source for the servo should be considered.

The servo and Pico must always share common ground.

### Lesson Learned

Motors and servos are electrically noisier and more demanding than logic components. Power stability matters.

## Summary

The main problems in this project were not caused by a single broken component.

Most issues came from:

- Understanding the build system
- Organizing files correctly
- Wiring components carefully
- Learning how breadboards work
- Understanding active-low button logic
- Managing GitHub authentication

These problems were useful because they helped me understand the full development process of an embedded project, from firmware and wiring to debugging and documentation.
